# LLM Dreams (Phase 2c-i) — Design

**Date:** 2026-06-19
**Status:** Approved (design); pending spec review

## Goal

Give the Slime Pet an LLM-written inner voice: on waking from a long sleep, it dreams a
short, calm line generated by a language model from the pet's derived context (environment +
relationship), with the existing template dreams as an always-on offline fallback. The
collectible-artifact mechanic stays exactly as it is.

## Motivation

Today dreams are assembled from fixed template fragments (`slime/dreams.py`). They're
charming but finite and impersonal. An LLM can weave the pet's weather/moon/presence/calendar/
inbox signals and its relationship (familiarity, recent journal tone) into fresh, soulful
dream lines — the heart of Phase 2c — while preserving the project's offline-first,
privacy-first ethos.

## Decisions (locked)

- **LLM location:** provider-agnostic. A server abstraction targets a **local Ollama** OR
  the **Anthropic API**, selected by config; with the on-device templates as fallback when
  the LLM is unreachable or unconfigured.
- **Personalization:** **relational** — the device sends a compact derived context
  (familiarity tier, 1–2 recent journal tone-words, artifact count, season) so dreams can
  nod to the bond and recent days. Only derived tone/buckets cross the boundary.
- **Artifacts:** unchanged, fully on-device. The LLM provides only the dream *narrative line*.
- **Privacy:** only derived buckets/tone-words reach the LLM; only the dream line returns.
  The prompt forbids inventing specific personal/identifying facts.

## Architecture

Two-tier, mirroring the oracle pattern. The home server owns the LLM call and the prompt;
the device sends a small context and shows whatever line comes back (or a template line).
Pure prompt/clean logic is host-tested; the provider HTTP clients are thin and exception-safe.

### Server: `server/app/llm.py` (new)

- `build_prompt(context) -> (system, user)` — **pure**. Turns the merged context dict into a
  system prompt (the slime-pet voice + hard constraints: ≤ DREAM_MAX_CHARS chars, 1–2 short
  sentences, calm ambient dream imagery, may gently reference the context, never invent
  specific personal facts, output the dream line only) and a user prompt listing the context
  tokens (weather_tag, moon_phase, season, coding_rhythm, day_load, inbox_load, familiarity
  tier, recent journal tone-words, artifact count).
- `clean_dream(text, max_chars) -> str` — **pure**. Collapse whitespace/newlines to single
  spaces, strip surrounding quotes, truncate to `max_chars` at a word boundary, ensure a
  terminal period. Returns "" for empty/garbage.
- `_ollama(system, user, url, model) -> str` and `_anthropic(system, user, key, model) -> str`
  — thin provider calls (httpx) returning raw text; each raises on failure (caller catches).
- `generate_dream(context) -> str | None` — selects the provider from `config.DREAM_PROVIDER`;
  returns `clean_dream(provider(...))` or `None` on unconfigured/empty/any exception.

### Server: `POST /dream` endpoint (in `main.py`)

- Accepts a small JSON body: `{"fam": int, "tones": [str, ...], "artifacts": int,
  "season": str}` (all optional; validated/clamped).
- Merges it with the server's current oracle signals (reuse the existing weather/moon/
  `_presence`/`_calendar`/`_inbox` derivations — same as `/oracle`) into one context dict.
- Returns `{"dream": generate_dream(context)}` when a non-empty line is produced; otherwise
  returns `{"dream": None}` (HTTP 200). The endpoint never 500s on LLM failure (device just
  falls back). Honors the same optional `ORACLE_TOKEN` bearer auth as `/oracle`.

### Server: config + compose

- `config.py`: `DREAM_PROVIDER` ("" | "ollama" | "anthropic"), `OLLAMA_URL`
  (default `http://localhost:11434`), `OLLAMA_MODEL` (default `llama3.2`),
  `ANTHROPIC_API_KEY`, `ANTHROPIC_MODEL` (default `claude-haiku-4-5`), `DREAM_MAX_CHARS`
  (default `120`).
- `docker-compose.yml` + `.env.example`: pass/document the new vars. **Both providers use the
  existing `httpx` dependency** — Ollama and Anthropic are both reached via their plain REST
  APIs, so no new server dependency (no `anthropic` SDK) is added.

### Device: `slime/netdream.py` (new adapter)

- `fetch(context) -> str | None` — POST `http://<ORACLE_HOST>/dream` with the context JSON
  (+ optional bearer token), parse `{"dream": ...}`, return the string or `None` on any
  failure/empty. Mirrors `slime/netoracle.py`; never raises into the loop.

### Device: pure context builder + `_dream_on_wake` wiring

- `slime/dreams.py` gains a pure `dream_context(tier, artifacts_mask, journal_records,
  season) -> dict` that derives `{fam, tones, artifacts, season}` — `tones` are 1–2 mood
  tone-words pulled from the most recent journal records (reusing the journal's mood mapping),
  `artifacts` is the popcount of the mask. Host-tested.
- `code.py` `_dream_on_wake`: build the context, call `netdream.fetch(context)`; if it
  returns a non-empty line, use it as the dream line; else fall back to the existing
  `dreams.generate(...)` line. The artifact roll, NVM artifact bookkeeping, sound, and
  `display.render_dream` are unchanged. (On battery wake the device is briefly awake; a few
  seconds for the LLM call is acceptable, with a short timeout → template fallback.)

## Data Flow

```
wake from long sleep ─> dreams.dream_context(tier, artifacts, journal, season) ─> {fam,tones,artifacts,season}
   └─> netdream.fetch ─POST /dream─> server merges with weather/moon/presence/calendar/inbox
         └─> llm.build_prompt ─> provider (ollama|anthropic via httpx) ─> llm.clean_dream
               └─> {"dream": line|None}
   device: line if present else dreams.generate(...) ─> display.render_dream(line, artifact_name)
   (artifact roll + NVM bookkeeping stay on-device, unchanged)
```

## Error Handling / Offline-First

- Unconfigured provider, LLM error, timeout, server down, or device offline → `netdream.fetch`
  returns `None` → device uses the template dream. Dreams ALWAYS work.
- `/dream` catches all LLM/provider exceptions and returns `{"dream": None}` (never 500).
- `clean_dream` guarantees a bounded, single-line, terminated string (or "" → treated as
  None upstream), so a rambling or oversized model output can't break the E-Ink render.
- Context fields are clamped/validated server-side; a malformed device body degrades to the
  server's own signals.

## Testing

**Server (CI `server` job, host pytest):**
- `build_prompt`: system prompt carries the constraints (char cap, "dream line only"); user
  prompt includes the provided context tokens (weather/season/fam/tones/etc.).
- `clean_dream`: collapses newlines/whitespace, strips quotes, truncates at a word boundary
  to `max_chars`, adds a terminal period, returns "" on empty.
- `generate_dream`: returns None when `DREAM_PROVIDER` is unset; returns the cleaned line
  when a provider function is monkeypatched to return raw text; returns None when the
  provider raises.
- `/dream`: returns `{"dream": <line>}` with a monkeypatched `generate_dream`; `{"dream": None}`
  when unconfigured; respects `ORACLE_TOKEN` (401 without it when set).

**Device (CI `check` job, host pytest):**
- `dreams.dream_context`: derives fam/tones/artifacts/season from inputs; tone-words from
  recent journal records; artifact popcount; empty journal → no tones.
- `netdream` is a thin adapter (device-only) — not host-imported; its fallback contract is
  exercised via `_dream_on_wake` reasoning (on-device verification).

**On-device:** with a provider configured, a long-sleep wake shows an LLM dream; with the
provider down/unset, it shows a template dream; latency stays acceptable on battery wake.

## Scope Guard (YAGNI)

- No long-term memory / persistence of past dreams (that is the next 2c sub-project).
- No streaming; one dream per wake.
- Artifacts unchanged; no new device NVM state.
- httpx for both providers (no `anthropic` SDK dependency) — call the REST APIs directly.
- Single home server; no multi-provider failover beyond "configured provider → template".
